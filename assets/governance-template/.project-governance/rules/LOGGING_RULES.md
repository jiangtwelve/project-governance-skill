# Logging Rules

开发记录必须轻量。判断标准：

> 新 agent 以后接手时，不知道这件事是否会导致需求、架构、流程或验收判断出错？会就记录，不会就不记录。

## 应该记录

- 需求范围确认或变化。
- 交互流程确认或变化。
- 架构、技术栈、数据模型、API 契约决策。
- 阶段验收结果。
- 影响后续开发的约束、风险、阻塞。
- 因用户反馈导致的返工方向。
- 治理规则升级、停用、重新启用。
- 已完成阶段和进入下一阶段的依据。

## 不应该记录

- 普通命令执行。
- “用户让我提交 git”这类无长期价值操作。
- 没有长期影响的临时调试。
- 格式化、重命名、无行为变化的小修。
- 纯实现细节，除非它改变架构或公共契约。
- agent 的过程性自言自语。

## 文件分工

- 当前状态：`ssot/PROJECT_STATE.md`。
- 需求变化摘要：`ssot/PRD.md`。
- 架构变化摘要：`ssot/ARCHITECTURE.md`。
- 接口变化摘要：`ssot/API_CONTRACT.md`。
- 决策原因：`decisions/INDEX.md` + 单个决策文件。
