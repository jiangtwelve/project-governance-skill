# Grilling Protocol

本协议内置 `grill-me` 的核心行为，不依赖外部 skill。任何 agent 即使没有安装 `grill-me`，也必须按本协议追问。

## 必须追问的情况

- 新项目、新功能、新版本、新需求。
- 需求、交互、架构、技术栈、数据模型、API 契约、部署方式变化。
- 阶段验收不通过后的修改方向。
- 用户反馈存在多种实现路线。
- agent 发现现有文档、代码、用户口述之间存在冲突。
- 任意会影响阶段推进、SSOT 或验收标准的决定。

## 可豁免追问的情况

以下事项通常不需要追问，但 agent 必须说明它们不影响 PRD、架构、API 和流程：

- 明确 bug 修复。
- 纯测试补充。
- 格式化、拼写修正、无行为变化的重命名。
- 不改变行为的依赖升级。
- 用户已经给出完整指令的小任务。

## 追问方式

- 一次只问一个关键问题。
- 每个问题必须给出推荐答案。
- 按决策依赖顺序追问，先解决上游问题，再进入下游细节。
- 如果问题可以通过读取项目文件解决，先读取文件，不要询问用户。
- 用户明确确认前，不得把有争议的假设写入 PRD、架构、API 或代码。

## 达成共识

只有用户明确表达以下含义时，才算确认：

- “确认”
- “同意”
- “按这个方案做”
- “验收通过”
- “进入下一阶段”
- 其他等价的明确授权

用户沉默、模糊赞同、agent 自行判断都不算确认。

## 追问后的落档

- 影响需求或交互：更新 `ssot/PRD.md`。
- 影响架构、数据模型、部署：更新 `ssot/ARCHITECTURE.md`。
- 影响接口：更新 `ssot/API_CONTRACT.md`。
- 影响阶段或下一步：更新 `ssot/PROJECT_STATE.md`。
- 有长期价值的原因和权衡：新增 `decisions/NNNN-title.md` 并更新 `decisions/INDEX.md`。
